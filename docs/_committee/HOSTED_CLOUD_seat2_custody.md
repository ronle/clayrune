## Seat 2 — Security & custody — RATIFY-WITH-CONDITIONS

**Decision:** RATIFY-WITH-CONDITIONS

**Summary (1-2 sentences):** The load-bearing "env-inheritance, zero dispatch change" BYOK claim is **TRUE** — the Claude dispatch spawns at `server.py:6938` (Mode B) and `server.py:7033` (Mode A) pass no `env=` and inherit `ANTHROPIC_API_KEY` from the parent process, so BYOK genuinely needs no dispatch code. But §5.3's "no plaintext at rest in the VM" and §10.1's "inject at wake, never persisted" are **both contradicted by code and by the locked sleep/wake mechanism**: an existing Settings surface writes provider keys plaintext to the persistent volume, the suspend snapshot holds the in-RAM key at rest where revoke can't reach it, and agent-driven output can leak the key into transcripts/Scribe on the volume. These are fixable in design; none requires relitigating BYOK or sleep/wake.

### Blockers (if any)

None that block the design outright — but **two conditions below are must-fix-in-design and gate Phase 1, and one gates the Phase 2 sleep/wake flip.** If the design ships its current §5.3/§10.1 promises verbatim to users (a custody guarantee in a ToS), that *would* be a blocker, because the guarantees are false as written. Downgrading from BLOCK to RATIFY-WITH-CONDITIONS on the basis that the promises are correctable spec edits, not architectural dead-ends.

---

### Ratifications

- **The env-inheritance claim is verified TRUE — this is the design's strongest load-bearing fact.** `ClaudeRuntime.dispatch` delegates to server.py via `_CLAUDE_HOOKS` (`agent_runtime.py:602-603`); the real spawn is `server.py:6938` (Mode B, `subprocess.Popen(cmd, stdin=PIPE, stdout=PIPE, ... cwd=pp, ...)` — **no `env=` kwarg**) and `server.py:7033` (Mode A — same, no `env=`). Python's `Popen` with `env=None` inherits the parent environment unchanged, so `ANTHROPIC_API_KEY` in the MC process env flows to the CLI with zero dispatch change. The Mode-A shared helper `_mode_a_dispatch` (`agent_runtime.py:2193-2209`) and the Gemini dispatch (`agent_runtime.py:1752-1768`) do `env = os.environ.copy()` then `env.update(env_extra)` — also inheriting, also consistent. §5's "nearly free" claim and §9's "Claude CLI dispatch — reused unchanged" are **correct**. Preserve this: any future refactor that switches dispatch to a curated `env=` dict silently breaks BYOK.

- **One-microVM-per-user is the right custody posture for arbitrary agent code + a live key (§4.1).** The single-instance invariant (`_check_port_conflict` at `server.py:13282`) maps cleanly onto one-VM-per-user, and hardware-grade Firecracker isolation is the correct answer to "the agent has a Bash tool and outbound internet and can read its own env." The blast radius of the agent exfiltrating its own key is the **user's own Anthropic account** — acceptable under an explicit trust model (see Condition 5).

- **Dropping per-VM mc-tunnel for hosted instances (§7) is a net custody *improvement*, not just a simplification.** Per `07-licensing.md` §2.2/§3.1, the proprietary `mc-tunnel` binary embeds a per-release shared `CLIENT_SECRET_PRIV` that §1.1 concedes is "theoretically extractable by a determined adversary." Removing that binary from the user VM removes a shared platform secret from a box running arbitrary agent code. Good. (Condition 6 makes this an explicit invariant rather than an accident.)

### Conditions (RATIFY-WITH-CONDITIONS only)

**Condition 1 (must-fix-in-design): The "no plaintext at rest in the VM" promise (§5.3) is false today — an existing Settings surface persists provider keys plaintext to the persistent volume.**
- **Why:** MC has `POST /api/agent/provider/<name>/env` (`server.py:3052`) which on every call does `_save_provider_env_file(data)` → writes `data/provider_env.json` **plaintext, `indent=2`** (`server.py:3008-3011`, path `PROVIDER_ENV_PATH = _DATA_ROOT / 'data' / 'provider_env.json'` at `:2995`). The allowlist explicitly accepts `_API_KEY` (`server.py:3071`). `_DATA_ROOT/data/` is exactly the directory §6 puts on the **persistent volume**. So a user who sets their key through the existing Settings → Agent Providers UI (which ships in the "reused unchanged" core) writes `ANTHROPIC_API_KEY` in cleartext to the durable volume — directly contradicting §5.3 and §10.1. Worse, the runtime precedence is split: `_hydrate_provider_env_into_os` only fills blanks (`k not in os.environ`, `server.py:3021`) so the *injected* key wins at boot, **but** the POST handler does `os.environ[key] = val_str` *unconditionally* (`server.py:3086`) — so a user setting the key at runtime both **overrides the orchestrator's injected key in the live process** and **persists it plaintext to the volume**. The injection model is silently defeatable by a surface the design forgot exists.
- **Proposed fix:** In hosted mode, **disable the `provider_env.json` write path for Anthropic credentials** (a hosted-mode guard on `agent_provider_set_env` that rejects/ignores `ANTHROPIC_*` and routes key changes to the control-plane vault instead), OR move `PROVIDER_ENV_PATH` to a tmpfs/secret-mount that is **not** part of the snapshotted volume. Also document that ANY `*_API_KEY`/`*_TOKEN`/`*_SECRET` set via this surface lands on the volume — Gemini/OpenAI BYOK keys have the same problem. Add a DATA_DIR/volume-exclusion note: `provider_env.json` must never be in a volume snapshot.
- **Gate phase:** Phase 1 (the first phase that injects a real key + onboards via the vault).

**Condition 2 (must-fix-in-design + soak-gate): The suspend snapshot holds the live key at rest, and §5.4 "revoke = clear vault + force restart" cannot reach a stale snapshot.**
- **Why:** The locked sleep/wake decision means a suspended microVM = a RAM+process snapshot persisted to disk (§4.2). The injected `ANTHROPIC_API_KEY` lives in the MC process memory (and in `os.environ`), so it is **in the snapshot blob at rest**. §10.1's "injected only at wake, never persisted" is undermined by the suspend mechanism itself, not by a code bug. §5.4's revoke ("clear vault entry + force a restart so the live process drops it") clears the *vault* and the *running* process — but a suspended VM has no running process to restart; its key is frozen in the snapshot. If revoke does not also **destroy the suspend snapshot**, a later resume-from-snapshot resurrects the revoked key into a live process, defeating revocation. (This compounds Seat 1's "resume-from-snapshot vs cold-boot" question — whichever wake path the economics choose, the snapshot path carries this key-at-rest exposure.)
- **Proposed fix:** (a) Require the provider's snapshot/volume to be **encrypted at rest** (confirm Fly Machines snapshot encryption in the Phase 0 spike; make it a provider-selection criterion in §4.2). (b) Redefine revoke (§5.4) as **clear vault + destroy any suspend snapshot + force cold-boot on next wake** — never resume a revoked VM from a pre-revoke snapshot. (c) State plainly in §10.1 that "never persisted in VM image" does **not** mean "never in a snapshot" — the honest guarantee is "encrypted at rest, and destroyed on revoke," not "never at rest."
- **Gate phase:** must-fix-in-design now; the snapshot-encryption + revoke-destroys-snapshot behavior is a **soak-gate** before Phase 2 (sleep/wake) flips on, since the exposure only exists once suspend is real.

**Condition 3 (must-fix-in-design): The "no key in logs" mitigation (§5.3/§10.1) holds only for argv and MC-internal explain-paths — NOT for agent-driven output, which lands in transcripts + Scribe on the volume.**
- **Why:** The mitigation is partially earned: the key travels via env, not argv, so it is not in the command line; and the agent_runtime explain-paths reference key names without printing values (confirmed — the brief's anchors at `agent_runtime.py:~3310/3444/3566/3716` are health-check/error strings). **But** the agent has a Bash tool. If it runs `env`, `printenv ANTHROPIC_API_KEY`, or `cat data/provider_env.json`, that output flows to stdout → `session['log_lines'].append(...)` verbatim (`server.py:4112` assistant text, `:4120` tool activity — **no secret scrubbing**) AND into the CLI's own `.jsonl` transcript, which §6 places on the persistent volume (`.claude/projects/` transcripts). Scribe then summarizes that transcript into `MEMORY.md` (also on the volume) — the CLAUDE.md memory pipeline reads "the CLI's on-disk `.jsonl` — the only full-fidelity source." So the key can reach: in-memory log buffer, on-volume transcript jsonl, and on-volume Scribe summary. This is the *user's own* key (blast radius = their account), so it is a confidentiality/log-hygiene issue, not a cross-tenant breach — but §5.3's flat "no key in logs" is false as written.
- **Proposed fix:** Scope the §5.3 claim honestly: "the key is not in argv and not printed by MC's own code paths; it CAN appear in agent transcripts if the agent dumps its own env, exactly as it can on a local BYO-machine install." Add an optional best-effort redaction pass (regex on `sk-ant-…` / the known key value) at the `log_lines.append` sites and before Scribe persists, posture matching Scribe's existing best-effort guards — never load-bearing. Note that this exposure is **identical to the existing local product** (where the key is also in the user's env and their own transcripts on their own disk); the *delta* is that in hosted mode the volume is **ours**, which is the real custody change (Condition 4).

**Condition 4 (must-fix-in-design): Reconcile the custody delta against the existing platform's "we never hold user data" foundation — the incident plan / ToS hook must be real, not hand-waved.**
- **Why:** The existing platform's foundational goal (`docs/remote-access/01-architecture.md` §1, line 20) is: **"MC-the-operator never holds user data: no dashboard contents, no project JSON, no agent transcripts, no hivemind state pass through or get stored on our infrastructure."** And `01-architecture.md` §1 lists **"Cloud-hosted agent execution"** as an explicit **v1 non-goal** (line 29). The hosted design inverts *both*: we now hold the codebases, the transcripts, the hivemind state, **and** the user's Anthropic key, on a volume we own. The custody delta vs. "already holding CF service tokens" is large and asymmetric: a CF service token is a *platform-scoped relay credential we mint and can revoke unilaterally at the edge*; an Anthropic API key is a *third-party billing credential whose compromise charges the user's account and which only the user (or Anthropic) can revoke*. §10.1 lists "clear ToS, incident plan" as a bullet but the design provides neither mechanism nor cross-reference. `07-licensing.md` §7 TOS skeleton predates hosted compute and says nothing about holding third-party API keys or storing user code/transcripts.
- **Proposed fix:** Add a §10.1 sub-section that (a) states the custody inversion explicitly against `01-architecture.md` §1's promise and marks that promise superseded for hosted mode; (b) defines a concrete incident plan: detection (vault access audit log), user notification SLA, and a **"rotate your Anthropic key" runbook** (we cannot revoke it for them); (c) requires a hosted-specific ToS clause covering custody of the API key + user code/transcripts before Phase 1 onboarding goes live. The key distinction to write down: CF tokens are platform-revocable; the Anthropic key is not — our only post-incident lever is "tell the user to rotate."
- **Gate phase:** must-fix-in-design; the ToS clause is a hard gate on Phase 1 (capturing the first real key).

**Condition 5 (must-fix-in-implementation): State the BYOK trust model explicitly and prove no shared control-plane secret is reachable from inside a user VM.**
- **Why:** The agent runs arbitrary code with a Bash tool and outbound internet (§10.5) under the *user's own* key — so the agent exfiltrating *its own* key is in-scope-acceptable (blast radius = the user's account; they consented by pasting it). The unacceptable failure is the agent reaching **beyond its own VM**: another user's VM, or a shared control-plane secret (the vault's KMS credential, the orchestrator's Fly API token, the ingress proxy's auth secret). One-VM-per-user (§4.1) plus dropping mc-tunnel's embedded `CLIENT_SECRET_PRIV` from hosted VMs (§7) is the right shape, but the design never enumerates "what secrets exist inside a user VM." The §9 "tiny in-VM agent so the orchestrator can read idle state" is a new control-plane↔VM channel — if it authenticates with a shared secret baked into the image, that secret is readable by the agent's Bash tool and becomes a cross-tenant pivot.
- **Proposed fix:** Add an explicit trust-model paragraph to §5/§10.1: "the user's key is exfiltratable by their own agent — that is the user's own account and is in-scope-acceptable; the invariant we defend is VM isolation and zero shared-secret reachability." Require: (a) the in-VM orchestrator agent (§9) uses a **per-VM, short-lived, non-shared** credential (or a pull model where the VM exposes read-only idle state and the control plane reads it, holding no inbound secret in the VM); (b) no orchestrator/Fly/KMS credential is ever present in the user VM's env or volume; (c) Phase 0 spike includes a check that `env` inside the VM contains *only* the injected `ANTHROPIC_API_KEY` and benign config — no platform secrets.
- **Gate phase:** Phase 1 (when the in-VM orchestrator agent + ingress are first built); the env-audit check belongs in the Phase 0 spike.

**Condition 6 (must-fix-in-implementation): Make "no shared platform secret in the hosted VM image" an explicit build invariant, not an emergent property of dropping mc-tunnel.**
- **Why:** §7 drops per-VM mc-tunnel/cloudflared for hosted instances, which removes the embedded `CLIENT_SECRET_PRIV` (`07-licensing.md` §2.2/§3.1) — but §9's "hosted mode flag" is described as merely skipping cloudflared/mc-tunnel *startup*. If the proprietary `mc_remote`/`mc_tunnel` artifacts are still **present on the image** (just not started), the embedded shared secret is still on a disk running arbitrary agent code, and a misconfiguration or a future code path could read it. The custody win of §7 must be enforced at image-build time, not left to a runtime flag.
- **Proposed fix:** The hosted VM image must **exclude the `mc_remote`/`mc_tunnel` binaries entirely** (not just skip starting them). Add this to the Phase 0 image-packaging step and assert it (no `CLIENT_SECRET`-bearing artifact on the image). This also cleanly preserves the open/closed boundary for hosted (Seat 4's concern) — hosted ships open MC core only.
- **Gate phase:** Phase 0 (image packaging) / Phase 1.

### Out-of-scope but flagged

- **(→ Seat 1) Snapshot-resume vs cold-boot is a shared crux.** My Condition 2 (key in snapshot) and Seat 1's in-flight-agent-across-suspend question both hinge on which wake path is real. If the economics (Seat 3) assume resume-from-snapshot to avoid MC boot latency, that is *exactly* the path that carries the key-at-rest exposure. The three seats' answers must be consistent: you cannot simultaneously promise "fast resume-from-snapshot," "never persist the key," and "cheap cold-boot."

- **(→ Seat 3) Egress is the key-exfiltration channel.** §10.5 outbound internet is required for the agent to function but is also how a compromised/prompted agent POSTs the key out. A per-VM egress allowlist (e.g., Anthropic API + GitHub + the user's declared endpoints) would shrink both the abuse surface (Seat 3) and the exfil surface (Seat 2). Flagging the overlap; the allowlist is one control serving both seats.

- **(→ Seat 4) The custody inversion changes the licensing/trust story.** `01-architecture.md` §1's "we never hold user data" was a *marketing and trust* pillar of the BYO-machine product. Hosted mode breaking it for hosted users (while the BYO product keeps it) means two different trust postures under one brand — Seat 4 should confirm the moat/positioning restatement (§7) accounts for "we hold your code + key now" being a real, disclosed difference, not glossed.

- **(observation, no owner) Multi-key surface.** BYOK is framed as Anthropic-only, but `provider_env.json` + the providers UI hold Gemini/OpenAI/Codex/AWS-Bedrock credentials too (the `SAFE_SUFFIXES` allowlist at `server.py:3071` includes `_PROFILE`/`_REGION`/`_CREDENTIALS` — AWS-shaped). If hosted users run non-Claude providers, every Condition above applies to those keys identically. The design should either scope hosted BYOK to Anthropic-only in v1 or extend the vault/injection/redaction story to all provider credentials.
