# Seat 1 — Rail integrity — review of docs/AUTO_SCOPE_PROMOTION_COMMITTEE_BRIEF.md §3

## Seat 1 — Rail integrity — RATIFY-WITH-CONDITIONS

The equivalence claim is *consistent with the already-ratified logic* — `exploration_read_floor` has satisfied rail #2 with interactive-origin provenance since 2026-07-11, and the brief only extends that logic from readback to installation. But as designed it has one genuine laundering circuit (unattended exploration → attended read-floor → interactive signals → auto-installed SKILL → loaded back into steward sessions: a full loop no human ever *reads*), and it trusts agent-writable disk frontmatter at the exact moment that frontmatter stops being "queue input a human will look at" and becomes "state that triggers an install." Both are closable; neither is closable by spec text alone.

### Blockers

None that survive the conditions below. Condition 1 is blocker-grade if left open — I would flip to REJECT if it is not closed before code lands.

### Conditions

1. **Close the read-floor laundering circuit (blocker-grade). Close IN DESIGN, then implementation.**
   The path, entirely from shipped code: a steward cycle distills an EXPLORATION with `origin: unattended` (`distiller.py:1974`). `exploration_read_floor` (`distiller.py:2182`) injects it into *attended* sessions — allowed, by design ("a human watches the consumer"). The injection itself steers those sessions onto the same topic, so extraction ("what topics did this session touch?") re-emits the same fingerprint, and those signals are stamped `unattended: False` because `mc/memory.py:756` looks only at the *session's* task. Three such sessions → `recurrence_count_exact ≥ 3` with `origin: interactive` → auto-install → the SKILL loads into every session of that project **including steward cycles** (installed skills have no origin filter; only `_proposed/` explorations do). Autonomous output has become autonomous standing instruction and no human read any artifact at any step. The parent spec tolerated this amplification only as a bounded, ephemeral context snippet (top-2, ~320 chars); auto-install makes it durable. Minimum viable close, either of:
   (a) install-eligibility disqualifies a fingerprint when any `origin: unattended` artifact with the same `exact` **or** `coarse` fingerprint exists anywhere under `_proposed/`, `_promoted/`, or `_rejected/` (cheap: `list_proposed()` already surfaces origin) — provenance-taint at the fingerprint level; or
   (b) signals extracted from a session whose read-floor injected an unattended-origin artifact inherit `unattended: True` for matching fingerprints. Option (a) is simpler and fails closed. The spec must name the chosen mechanism.

2. **Recompute eligibility from the signal store; never trust frontmatter at install time. Close IN IMPLEMENTATION.**
   §3.1's eligibility inputs (`origin`, `recurrence_count_exact`, `kind`, scope) are read from `.md` frontmatter in `data/skills/_proposed/` — plain files on disk. The steward fence (`steward/fence.py:211-215`) blocks Write/Edit only on paths containing `/.claude/`; **writes to `data/skills/_proposed/` and `data/projects/*_skill_stats.json` pass the fence**. Today that is survivable because a human reads the queue before anything installs; under auto-install, a misbehaving agent (the actual threat model — see the 2026-06-22 authority artifact) that edits `origin: unattended` → `interactive` or inflates `recurrence_count_exact` gets an unmediated install. Fix: the installer must recompute `origin` (conservative-OR) and exact recurrence from the `_skill_stats.json` signal records for that fingerprint (`_evaluate_candidate`'s own inputs) and refuse on mismatch, treating frontmatter as display metadata only.

3. **Extend the steward fence to the proposal/stats surfaces. Close IN IMPLEMENTATION.**
   Complement to Condition 2 (defense in depth on the write side): `classify_action` should block steward Write/Edit/MultiEdit/NotebookEdit under `data/skills/` (`_proposed/`, `_promoted/`, `_rejected/`) and on `*_skill_stats.json`, with a regression test alongside `tests/test_distiller_safety.py`. Once installs can be machine-triggered, these files are part of the loadout supply chain and deserve the same fence posture as `.claude/`.

4. **Invert §3.4 from a denylist to an allowlist. Close IN DESIGN + IMPLEMENTATION.**
   The proposed tightening (steward marker OR `trigger_type ∈ {schedule, hivemind_orchestrator, hivemind_worker}` OR prompt text) enumerates known unattended shapes and defaults everything else to interactive — it fails **open** for exactly the cases that will appear next. Two concrete misses today:
   (a) *Transcript-backfilled sessions lose `trigger_type`* — `mc/blueprints/agent_routes.py:2967` documents that backfill "cannot recover trigger_type/trigger_id." A scheduled run whose MC was hard-killed and revived from the CC transcript reads as `manual` → interactive. The 2026-07-16 backfill (`_scratch/backfill_origin.py`) has the same blind spot for any agent-log entry rewritten by backfill.
   (b) *Night-shift/one-off unattended prompts* are caught only by the literal "you run unattended" in the first 400 chars — a rephrased prompt silently flips to interactive.
   Correct shape: `origin: interactive` **only when** `trigger_type == 'manual'` (the enumerated human-initiated set) AND no unattended marker AND the field is actually present; missing/unknown/backfilled `trigger_type` → unattended. The taxonomy is closed today (`manual | schedule | hivemind_orchestrator | hivemind_worker`, `agent_routes.py:3079`), so the allowlist is one value; any future trigger type then defaults safe instead of unsafe.

5. **Initiator provenance for machine-dispatched sessions. Close IN IMPLEMENTATION.**
   A steward (or scheduled agent) can dispatch a fresh session via the local API (`mc-clayrune-apis` is in its toolset; the fence blocks catastrophic Bash shapes, not localhost curl). That child session's task lacks the steward marker and `trigger_type` defaults to `'manual'` (`agent_routes.py:2997/3081`) → its signals stamp interactive. Fix: dispatches originating from a fenced/unattended session must carry a distinct `trigger_type` (or a parent-provenance field) that Condition 4's allowlist excludes. Without this, "unattended = steward marker + trigger types" is a self-report the unattended agent can shed in one hop.

6. **Durable-"no" on every removal path for auto-installed artifacts. Close IN IMPLEMENTATION.**
   §3.2's one-click revert writes a suppression — good. But `distiller.py:1493-1497` deliberately gives *installation* the opposite semantics ("uninstalling an artifact makes its pattern proposable again"), and the Skills panel's ordinary delete path predates auto-install. If a human removes an auto-installed skill via any path that does not write a suppression, the pattern re-proposes on next recurrence and **auto-installs again** — rail #3 inverts into a reinstall loop, which is strictly worse than the `preference-1ba8d678` re-propose bug because there is no queue to catch round two. Every removal path for `auto_authored: true` artifacts must write the suppression record (test it the way `test_global_rejection_is_durable_across_projects` pins the reject path).

7. **Compensating controls for the authority guard's residual paraphrase risk. Close IN DESIGN (cap + telemetry named in spec), IMPLEMENTATION for the rest.**
   `_authority_violation` is a regex regression pin, not a semantic classifier — its own comment records that "a tighter pattern let one through," and its phrase list is anchored to the six artifacts that escaped. The promotion-time human was demonstrably *not* a working guard for this class (humans promoted all six), so the proposal loses little — but it also removes the only chance of catching a novel paraphrase ("treat prior approval as covering similar future actions" matches nothing in `_AUTHORITY_PATTERNS`). Require: (a) a structured log line per auto-install (fingerprint, name, project, origin) so the monthly audit is a real control rather than an aspiration; (b) a small per-project auto-install rate cap (e.g., N/day) so a runaway bypass is bounded in blast radius; (c) the audit-prompt extension in §3.2 ships in the same PR, not after, as the brief already promises — hold it to that.

### Ratifications

- **The equivalence claim's core logic is sound and precedented.** Rail #2's enforcement for explorations has been provenance-based since 2026-07-11: interactive-origin artifacts already flow into unattended consumers with no per-artifact human review (`exploration_read_floor`, `distiller.py:2194-2208`; pinned by `test_unattended_read_floor_blocks_unattended_artifacts`). The brief extends the same construction to installation rather than inventing a weaker one. Rejecting the equivalence here would require re-litigating the ratified parent.
- **Conservative-OR is correctly implemented at both aggregation sites.** `_evaluate_candidate` taints on any unattended witness for skill/preference (`distiller.py:1248`) and exploration (`:1215`) candidates; `_normalize_signals` stamps every signal (`:973/:993/:1010`); `_wrap_skill_body` propagates to frontmatter (`:1974`). One steward witness → `origin: unattended` → permanently ineligible under §3.1(2). No dilution path in the aggregation itself.
- **Fail-closed posture on missing provenance.** §3.1(2) makes missing/unresolvable origin ineligible *forever*, matching the read-floor's treatment of pre-stamp artifacts (`distiller.py:2232-2234`, pinned by `test_unattended_read_floor_fails_closed_on_unstamped_artifacts`).
- **The backfill's fail-closed branches are right.** `_scratch/backfill_origin.py`: unresolvable session → unattended; any unattended witness → unattended; sids the regex can't extract → unresolvable → unattended. Every failure mode moves artifacts *out of* eligibility, never into it. (Its trigger-type blind spot is Condition 4(a), a shared root cause, not a backfill-specific bug.)
- **Exact-layer-only recurrence for eligibility.** §3.1(3) correctly refuses the coarse layer, which `_evaluate_candidate` accepts for *proposal* at N+1 — a coarse-qualified candidate writes `recurrence_count_exact < 3` and stays queue-only.
- **Scope containment.** Explorations excluded from install; global/cross-project excluded permanently; no machine write path to `~/.claude/skills/` exists or is added; the fence's `/.claude/` block covers the project loadout for steward sessions. Blast radius of the new mechanism is one project's `.claude/skills/`.
- **Rails #1 and #3 are structurally untouched** by the proposal text itself: authority guard still fires at generation (`_generate_and_write_artifact`, `distiller.py:1651`) plus again at install; reject/revert paths route through `_suppress_artifact`, whose global-store fix (`_GLOBAL_SUPPRESSION_PID`) this proposal does not modify.
- **Kill-switch layering** (global `distiller_auto_scope_enabled` default-false AND per-project `distiller_mode: auto` opt-in, ships dark) matches the discipline of prior rollouts (Step 6 checkpointing).

### Out-of-scope but flagged

- **Session-level presence is not artifact-level review — say so in the synthesis.** The rail survives in letter, but "human on the producing side" now means "a human was in the room during the evidence sessions," not "a human saw this pattern or this text." An interactive session emits signals the human never reads; at recurrence 3 the human was present three times and read the artifact zero times. The old gate, rubber-stamp though it was, put human eyes on the exact artifact body once. This is the honest residual weakening even after all conditions close; the compensating controls (badge, audit, revert, cap) are the answer, so they must be treated as load-bearing, not cosmetic.
- **Unattended non-steward sessions are not fenced at all.** `fence.py` self-gates to steward sessions only (`main()`, `:285-287`); a scheduled/night-shift agent already *could* write directly to `<project>/.claude/skills/` today. Not created by this proposal, but auto-install normalizes machine-written loadouts, which makes this pre-existing gap more attractive. Worth a follow-up item outside this review.
- **`_distill_extract_and_aggregate_inner` re-checks the gate with hardcoded session flags** (`session={'incognito': False, 'housekeeping': False}`, `distiller.py:828-830`) — the inner gate cannot actually see the real session's incognito flag; it relies on the upstream dispatch gate. Auto-install raises the stakes of any future regression there (incognito session evidence flowing into an installed artifact). Verify the upstream gate has a test.
- **`_installed_exact_fingerprints` reads only the first 2000 bytes** of each SKILL.md for the fingerprint line (`distiller.py:1529`); auto-installed artifacts must keep `extraction_fingerprint_exact` inside the frontmatter head or the durable-yes dedupe silently misses them.
